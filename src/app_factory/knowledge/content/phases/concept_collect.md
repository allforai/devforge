# Concept Collection Knowledge

## Purpose

Concept collection is not a fixed questionnaire. It is a dynamic discovery phase that converts vague intent into a structured concept model suitable for planning.

## What This Phase Must Produce

At minimum, concept collection should produce:

- problem framing
- target users or player archetypes
- domain-specific constraints
- success definition
- scope boundaries
- concept risks
- assumptions worth testing

## Collection Rules

- Prefer one batch of structured questions over repeated interruptions.
- Ask only for information that materially changes planning.
- When uncertainty is acceptable, record an assumption and move forward.
- When a decision is expensive to reverse, escalate it.

## Project-Specific Behavior

Concept collection must branch by archetype:

- game projects ask about fantasy, loops, world, style, and balance expectations
- ecommerce projects ask about flows, roles, payment, stock, and state transitions
- tools/SDK projects ask about users, integration surfaces, API contracts, and developer ergonomics

## Output Shape

Concept collection should end in a structured concept artifact, not only chat history. The output should be sufficient for:

- role activation
- domain graph synthesis
- work package planning
- acceptance criteria seeding
