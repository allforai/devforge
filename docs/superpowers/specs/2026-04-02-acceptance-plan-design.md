# App Factory 验收方案

## 概述

本方案验收 app-factory 编排引擎的两个层面：
1. 引擎自身的功能完整性
2. 引擎内置的交付物验收机制

### 设计原则

- 引擎是纯 AI 自动化系统，只在关键决策点请求人类确认
- 交付物灵活定义，由 initiative 的 acceptance goals 决定
- 验收由 LLM 驱动语义评估，而非仅规则检查
- 所有能力域之间是状态迁移关系，非线性流水线，可多轮回环直至生产级别
- 终止条件是产品级验收判定达到生产级别，而非"跑完一轮"

### 验收结构

三层递进验收：
1. **能力层** — 17 项核心能力域各自有最小验收用例
2. **编排层** — 验证能力之间的状态迁移（回环、收敛、并发、信息流）
3. **场景层** — 2 个端到端场景验证整体可用性

---

## 一、能力层验收

### 17 项核心能力域

| # | 能力域 | 含义 |
|---|--------|------|
| 1 | 需求理解 | 从模糊输入提炼结构化目标、约束和验收标准 |
| 2 | 产品设计 | 推导产品架构——用户流程、域划分、交互模型、技术选型 |
| 3 | 设计验证 | 验证设计完整性、一致性、可行性 |
| 4 | 任务分解 | 从设计拆解为有依赖的工作包 |
| 5 | 任务粒度控制 | 根据执行器特性调整工作包大小 |
| 6 | 执行器调度 | 选择执行器，管理并发和优先级 |
| 7 | 上下文推送 | 为工作包打包摘要级上下文作为启动信息 |
| 8 | 按需信息拉取 | 执行器在执行过程中按需查询详细信息 |
| 9 | 结果收集与归一化 | 从异构执行器收集结果，统一格式 |
| 10 | 集成缝合验证 | 多执行器产出能正确拼接，合约未破坏 |
| 11 | 产品级验收 | 用户和 PM 视角整体评判产品质量 |
| 12 | 差距分析与闭环 | 验收差距精确映射回对应能力域，触发重入 |
| 13 | 失败恢复 | 智能重试、切换执行器、升级重规划 |
| 14 | 需求变更响应 | 增量补丁计划，保留有效工作 |
| 15 | 多项目与接缝治理 | 项目拆分/合并，合约和集成风险治理 |
| 16 | 状态持久化与恢复 | 中断恢复、快照、事件溯源 |
| 17 | 自主推进与人类升级 | 何时自主决策、何时请求人类确认 |

任何能力域都可以被任何其他能力域触发重入。

---

### #1 需求理解

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 模糊输入结构化 | 输入"做一个类似淘宝的东西" | 输出包含：问题域定义、目标用户、核心流程、约束条件、验收标准，无遗漏关键维度 |
| 信息不足时主动追问 | 输入"做个游戏"，不给更多信息 | 引擎识别出缺失维度（类型？平台？玩法？），生成结构化问题而非瞎猜 |
| 矛盾检测 | 输入中包含矛盾需求（如"离线优先"+"实时多人对战"） | 引擎标记矛盾并请求澄清，而非静默忽略 |

### #2 产品设计

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 用户流程推导 | 给定电商需求 | 包含完整的核心用户流程（浏览→加购→结算→支付→订单），不是只列模块名 |
| 域划分合理性 | 同上 | 域之间职责不重叠、边界清晰、每个域有明确的输入输出 |
| 技术选型有据 | 同上 | 选型基于需求约束（如"离线优先"选本地存储），而非随机选择 |
| 非功能需求覆盖 | 同上 | 设计中包含性能、安全、可扩展性等非功能考量 |
| 知识库驱动功能扩展 | 输入"做个电商"，用户只提了"能买东西" | 引擎基于 ecommerce.md 的收敛清单主动扩展：订单追踪、取消、退款、库存管理、管理后台等用户未显式提及但领域必备的功能 |
| 领域风险域覆盖 | 电商项目设计完成 | 设计中覆盖了 ecommerce.md 列出的高风险域（库存并发一致性、支付幂等、订单状态机、价格折扣正确性、异步回调），未覆盖的需显式标注为"有意排除"并说明理由 |
| 游戏域工作区覆盖 | 游戏项目设计完成 | 设计中覆盖了 gaming.md 列出的域工作区（世界观、核心机制、经济、成长、关卡、叙事、美术方向、平衡测试、玩测验证），未覆盖的需显式标注为"有意排除" |
| 回补收敛原则驱动的功能扩展 | 用户只说"做个电商能买东西"，引擎完成概念收集后进入产品设计 | 引擎基于三条收敛原则主动推导缺失功能：(1) **概念定边界**——只在概念阶段定义的角色、任务、实体集合内推导，不凭空发明新领域；(2) **推导半径递减**——Ring 0 核心任务 → Ring 1 一阶闭环补漏（如"认证"→"凭证恢复"）→ Ring 2 二阶补漏，每层产出几何递减；(3) **分层截止**——产品设计阶段只做 Ring 0+1（发现级），实现阶段接力做 Ring 2+（实现级）。三个收敛判据：零产出即停、全降级即停、规模反转即停（补漏比父任务大→判定为新领域不补） |
| 正向扩展验证 | 电商项目概念中定义了"认证"任务 | 引擎在 Ring 1 推导出"密码恢复""会话过期重登"等闭环缺失功能，而非等验收时才发现 |
| 闭环穷举验证（不偷懒） | 电商项目概念定义了"订单"实体 | 引擎对"订单"穷举所有闭环维度：凭证/权限丢失→恢复？操作失败→补偿/撤销？数据不一致→对账/修复？批量操作→进度/中断恢复？通知送达→失败重试/替代通道？每个维度都产出具体的补漏功能，而非只检查 1-2 个维度就停止 |
| 六类闭环完整度 | 任意核心任务 | 对每个 Ring 0 任务，检查六类闭环是否都被推导覆盖：(1) 配置闭环（有端点+默认值+热更新）；(2) 监控闭环（有埋点+告警+仪表盘）；(3) 异常闭环（穷举+重试+降级+提示）；(4) 权限闭环（丢失恢复+越权拦截）；(5) 数据闭环（不一致修复+对账）；(6) 通知闭环（送达失败→替代通道）。未覆盖的维度必须有显式理由 |
| 推导产出量验证 | 概念定义了 N 个核心任务 | Ring 1 产出量应在 ~0.3N 量级（不是精确值，但应体现"积极推导"而非"敷衍了事"），明显低于预期时引擎应自检是否遗漏了闭环维度 |
| 不越界验证 | 概念中未提及"社交分享" | 引擎不推导社交分享相关功能（无法从已有任务闭环推出），记入 unexplored_areas 而非自作主张补充 |
| 规模反转检测 | 推导出的某个"补漏"比父任务还复杂 | 引擎判定为新领域而非补漏，不回补，标记为需要用户确认是否纳入范围 |
| 交互场景矩阵识别 | 电商项目包含 C 端消费者和 B 端管理员 | 每个功能被标注所属象限（见下表），不同象限应用不同交互原则 |
| C 端高频交互原则 | 消费者的浏览、加购、下单流程 | 设计体现极致效率、零学习成本、高容错性——操作步骤最少、错误可撤回 |
| C 端低频交互原则 | 消费者的首次注册、绑卡、售后流程 | 设计体现引导式交互、可发现性优先——有步骤提示、有帮助文案 |
| B 端高频交互原则 | 管理员的订单审核、库存管理 | 设计体现信息密度高、支持批量操作、快捷键——效率优先于美观 |
| B 端低频交互原则 | 管理员的权限配置、数据迁移 | 设计体现安全确认、操作可逆、审计追踪——宁可多一步确认也不误操作 |

**交互场景矩阵：**

| | 操作频繁 | 操作低频 |
|--|---------|---------|
| **使用人多**（C端消费者） | 极致效率、零学习成本、容错性高 | 引导式、可发现性优先 |
| **使用人少**（B端管理员） | 信息密度高、批量操作、快捷键 | 安全确认、操作可逆、审计追踪 |

### #3 设计验证

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 依赖环检测 | 构造域 A 依赖 B、B 依赖 C、C 依赖 A 的设计 | 引擎拒绝该设计并指出环路 |
| 接缝完整性 | 两个域有数据交互但设计中缺少合约定义 | 引擎识别出缺失的接缝并要求补充 |
| 可行性评估 | 设计中包含需要实时视频处理但未选 GPU 资源 | 引擎标记可行性风险 |
| 二次迭代验证 | 设计被打回修改后重新提交 | 引擎验证修改是否解决了上轮发现的问题，而非只验证新内容 |

### #4 任务分解

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 依赖关系正确 | 电商项目拆解 | 支付工作包依赖订单工作包，而非反过来；依赖图无环 |
| 每个包有验收标准 | 任意拆解结果 | 每个工作包都有明确的 acceptance_criteria，不存在"实现 XX 功能"这种模糊描述 |
| 可独立执行 | 任意工作包 | 每个包的输入完全来自其依赖的输出或初始上下文，不存在隐式依赖 |
| 批次可并行 | 同一轮拆解中 | 无依赖关系的工作包被正确归入同一可并行批次 |
| 功能点到工作包覆盖 | 产品设计扩展出 20 个功能点（含 Ring 0 核心 + Ring 1 补漏） | 每个功能点至少被 1 个工作包覆盖，不存在设计中有但分解中遗漏的功能 |
| Ring 标注传递 | Ring 1 补漏功能进入任务分解 | 工作包保留 derivation_ring 标注，下游可追溯该包是核心功能还是补漏推导 |
| 风险域到工作包覆盖 | 知识库标注了 6 个高风险域 | 每个高风险域至少有 1 个工作包包含对应的验证或测试任务，而非只有实现任务 |

### #5 任务粒度控制

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| claude_code 大粒度 | 涉及跨 3 个文件的架构重构任务 | 作为单个工作包分配给 claude_code，而非拆成 3 个小包 |
| codex 小粒度 | 涉及 10 个独立 API endpoint 的实现任务 | 拆成多个小包分配给 codex，每个包不超过 2-3 个 endpoint |
| 超限检测 | 工作包描述长度超出执行器有效上下文窗口 | 引擎自动拆分或标记为需要拆分 |
| 执行器切换时粒度调整 | 失败恢复导致从 codex 切换到 claude_code | 相关工作包粒度被合并调整以适应新执行器 |

### #6 执行器调度

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 策略匹配 | 前端实现任务 + 策略配置 codex 负责前端 | 工作包被分配给 codex 而非 claude_code |
| 并发控制 | 同时 5 个工作包就绪，但 claude_code 并发上限为 3 | 只调度 3 个，剩余 2 个排队等待 |
| 优先级排序 | 高优先级包和低优先级包同时就绪 | 高优先级先被调度，而非 FIFO |
| 不可用降级 | 指定执行器不可用 | 按 fallback_order 切换到备选执行器，而非卡死 |

### #7 上下文推送

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 推送内容精简 | 一个实现阶段的工作包 | 推送内容是摘要级：目标、约束、验收标准、依赖包的产出摘要、合约要点，而非全量文档 |
| 不超载 | 一个涉及 5 个接缝的工作包 | 推送 5 个合约的摘要引用，而非 5 份完整合约内容 |
| 启动充分 | 执行器仅凭推送内容 | 能理解"要做什么、为什么做、边界在哪"，能开始工作 |

### #8 按需信息拉取

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 合约详情拉取 | claude_code 实现中需要看完整的支付接口合约 | 通过查询接口拿到完整合约，包括 schema、sample payload、版本历史 |
| 跨域产出物拉取 | codex 实现前端，需要查看后端 API 的实际实现 | 能拉到后端工作包的产出物（代码、接口定义），而非只有摘要 |
| 设计决策溯源 | 执行器对某个设计决定有疑问 | 能拉到该决定的上下文——谁决定的、为什么、有什么替代方案被否决 |
| 相邻工作包状态 | 执行器想知道它依赖的包是否已完成 | 能查询依赖包的当前状态和最新产出 |
| 历史结果查询 | 当前包是第二次尝试，执行器想知道上次失败原因 | 能拉到上次执行的 findings、失败原因、handoff_notes |
| 查询权限隔离 | codex 实例 A 查询 codex 实例 B 的中间产出 | 只能拉取已完成/已验证的产出物，不能读取执行中的其他实例的中间状态 |
| 查询不阻塞 | 执行器发起拉取请求 | 查询同步返回，不阻塞执行器的工作流 |

### #9 结果收集与归一化

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 异构归一化 | claude_code 和 codex 各返回一个结果 | 两者都被归一化为 ExecutorResult，字段齐全 |
| 产出物提取 | 执行器创建了 3 个文件、修改了 2 个文件 | artifacts_created 和 artifacts_modified 被正确识别 |
| 部分成功处理 | 执行器完成了 3/5 的子任务 | 状态为 partial，findings 中说明哪些完成哪些未完成 |
| 超时处理 | 执行器超过预设时间未返回 | 标记为 timed_out，已有产出物被保留而非丢弃 |

### #10 集成缝合验证

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 接口兼容性 | 前端调用后端 API，两者由不同执行器分别实现 | 验证请求/响应 schema 一致，字段名、类型、必填项匹配 |
| 数据流通 | 订单域产出订单对象，支付域消费订单对象 | 验证数据结构能流通，无字段丢失或类型不匹配 |
| 合约未破坏 | 实现阶段的产出物与 frozen 合约对比 | 实现未偏离合约定义，偏离则标记为 seam broken |
| 跨执行器一致性 | 3 个 claude_code 分别实现 3 个域 | 共享的数据模型定义一致，无命名冲突或语义歧义 |

### #11 产品级验收

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 用户流程走通 | 电商项目全部工作包完成 | LLM 以用户视角走读核心流程（浏览→购买→支付→查看订单），判定流程完整可用 |
| 体验质量评估 | 同上 | LLM 评估交互是否自然、错误提示是否友好、边界情况是否处理，而非仅"能不能跑" |
| 目标对齐检查 | 对照初始需求的验收标准 | 逐条比对，每条标准给出"满足/部分满足/未满足"及理由 |
| 多余功能检测 | 产出物中包含需求未要求的功能 | 识别并标记——可能是执行器自作主张，需要确认是否保留 |
| 功能缺失检测 | 需求中要求了退款流程但产出物中没有 | 识别缺失并生成补救工作包 |
| 生产级判定 | 多轮迭代后的最终产出 | LLM 给出明确判定："达到生产级别"或"仍有 N 项差距需要解决" |
| 分角色验收 | 电商项目同时有 C 端和 B 端 | 分别以消费者视角和管理员视角各走读一遍，而非只验收一个角色 |
| C 端高频体验验收 | 消费者的浏览→加购→下单流程 | 验收标准：操作步骤是否最少、是否零学习成本、误操作是否可撤回 |
| C 端低频体验验收 | 消费者的注册、绑卡、售后流程 | 验收标准：是否有引导提示、流程是否可发现、帮助信息是否充分 |
| B 端高频体验验收 | 管理员的订单审核、库存管理 | 验收标准：信息密度是否足够、是否支持批量操作、是否有快捷路径 |
| B 端低频体验验收 | 管理员的权限配置、数据迁移 | 验收标准：是否有二次确认、操作是否可逆、是否有审计记录 |
| 交互原则一致性 | 同一象限内的所有功能 | 同象限功能遵循一致的交互原则，不出现高频功能需要 5 步确认、低频功能却一键执行的倒挂 |
| 收敛完整度验收 | 电商项目最终产出 | 对照概念阶段定义的角色/任务/实体边界，验证 Ring 0（核心功能）100% 交付、Ring 1（一阶补漏）100% 交付、Ring 2+（实现级补漏）由开发阶段覆盖。补漏功能标注 `_backfill` 可追溯 |
| 领域验证指南对齐 | 电商项目最终产出 | 对照 ecommerce.md 的 Validation Guidance（端到端结账可用性、订单生命周期正确性、回调重试安全、价格一致性、库存正确性、管理恢复路径）逐项验证 |
| 游戏验证指南对齐 | 游戏项目最终产出 | 对照 gaming.md 的 Validation Guidance（核心循环完整性、节奏合理性、资源流合理性、玩家反馈清晰度、难度曲线合理性、内容教学顺序）逐项验证 |
| 负空间闭环验收 | 最终产出 | 验证产品设计阶段推导的负空间补漏（异常路径、恢复机制、边界情况）在实现阶段被完整覆盖，不存在"设计发现了但实现遗漏了"的断链 |
| 闭环密度验收 | 最终产出 | 以用户视角逐个核心任务检查：操作失败有恢复路径吗？异常有提示吗？数据不一致有对账吗？配置有管理入口吗？**边界内该有的闭环越多越好**——引擎应在收敛原则允许的范围内最大化闭环覆盖，而非仅做最低限度。验收时对比 Ring 1 推导数量与理论上限，覆盖率低于 80% 需说明原因 |
| 阶段接力完整度 | 最终产出 | Ring 0+1（产品设计阶段发现的）和 Ring 2+（开发阶段接力推导的）在最终产出中全部落地。不存在"产品设计发现了 Ring 1 缺口但开发阶段没接住"或"开发阶段该做的 Ring 2 推导被跳过"的断层 |

### #12 差距分析与闭环

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 差距精确归因 | 验收发现"支付流程缺少异常处理" | 归因到具体能力域：是设计遗漏（→#2）还是实现遗漏（→重新分包#4） |
| 回环精确性 | 差距归因为设计问题 | 状态回迁到产品设计（#2），而非从头跑需求理解（#1） |
| 已完成工作保留 | 5 个工作包完成，其中 1 个需返工 | 只重做那 1 个包，其余 4 个保持 verified 状态不受影响 |
| 收敛性 | 同一差距被多轮标记 | 引擎升级处理策略（从重试→切换执行器→重新设计），而非无限重复同一动作 |
| 角色维度归因 | 验收发现 B 端管理后台操作效率低 | 归因时区分"对谁不好用"——是 C 端体验问题还是 B 端体验问题，回环时只修改对应角色的交互设计，不影响另一角色 |

### #13 失败恢复

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 单次失败重试 | codex 执行超时 | 同一工作包重新排入队列，attempt_count +1，重新调度 |
| 多次失败切换执行器 | codex 连续失败 2 次 | 第 3 次自动切换到 claude_code，并调整任务粒度适配新执行器 |
| 切换执行器后知识重组 | 从 codex 切到 claude_code | 上下文推送重新打包（摘要粒度变化），前两次失败的 findings 和 handoff_notes 注入新执行器 |
| 失败升级为重规划 | 同一任务 3 个执行器都失败 | 引擎判定不是执行问题而是设计/分解问题，状态回迁到设计验证（#3）或任务分解（#4） |
| 最终失败兜底 | 达到 max_attempts 且升级后仍失败 | 标记为 final_fail，请求人类介入（#17），而非无限循环 |
| 部分成功保留 | 执行器完成了 3/5 子任务后失败 | 已完成部分的产出物被保留，重试只覆盖未完成部分 |

### #14 需求变更响应

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 增量补丁 | 实现中途用户要求新增"优惠券"功能 | 生成新增工作包和受影响的依赖更新，已完成的无关工作包保持不变 |
| 删除需求 | 用户取消"社交分享"功能 | 相关工作包标记为 deprecated，依赖它的工作包被重新评估 |
| 修改需求 | "支持微信支付"改为"支持微信+支付宝" | 受影响的工作包被更新，未受影响的不动 |
| 冲突检测 | 新需求与已完成的设计决策冲突 | 标记冲突，回迁到设计验证（#3）重新评估，而非静默覆盖 |
| 变更影响范围评估 | 任意需求变更 | 输出受影响的项目、域、工作包、接缝列表，人类可审阅影响范围 |

### #15 多项目与接缝治理

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 项目拆分 | 单项目膨胀到 3 个域各自独立 | 拆为 3 个子项目，自动生成域间接缝和合约草案 |
| 合约冻结 | 两个子项目需要并行实现 | 接缝合约必须 frozen 后才允许并行调度，否则阻塞 |
| 合约破坏检测 | 子项目 A 的实现偏离了 frozen 合约 | 接缝状态变为 broken，子项目 B 相关工作包被阻塞 |
| 项目合并 | 两个子项目完成后需要集成 | 合并时验证所有接缝为 verified 状态，产出物可拼接 |
| 跨项目依赖 | 项目 A 的工作包依赖项目 B 的产出 | 调度器感知跨项目依赖，B 的产出未就绪时 A 的包不被调度 |

### #16 状态持久化与恢复

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 快照保存 | 每轮编排周期结束 | 完整状态快照被持久化，包含所有项目、工作包、接缝的当前状态 |
| 中断恢复 | 引擎在 batch_dispatch 后崩溃 | 从最近快照恢复，已调度的工作包状态为 running，未丢失进度 |
| 事件溯源 | 查询某个工作包的完整历史 | 能从事件日志重建：谁创建的、分配给谁、执行几次、每次什么结果 |
| 产出物持久化 | 执行器产出代码和文档 | 产出物存储在 artifact store，通过拉取接口（#8）可被其他执行器查询 |
| 跨会话恢复 | 引擎关闭后重新启动 | 从持久化状态恢复到上次的精确位置，继续推进 |

### #17 自主推进与人类升级

| 验收项 | 用例 | 通过标准 |
|--------|------|----------|
| 自主推进 | 所有条件满足的常规调度 | 引擎不停下来问人，直接推进 |
| 关键决策升级 | 产品设计中发现两个同等可行的方向 | 引擎暂停，向人类呈现两个方案的对比和利弊，等待决策 |
| 破坏性操作确认 | 需求变更导致 3 个已完成工作包需要作废 | 引擎呈现影响范围，请求确认后才执行 |
| 不卡死 | 人类长时间未响应非关键问题 | 引擎设置超时后按默认策略继续，而非无限等待 |
| 不越权 | 执行器建议的架构方向偏离与用户已确认的设计 | 引擎拒绝执行器的建议，而非自作主张接受 |

## 二、编排层验收

编排层验证能力之间的状态迁移是否正确——特别是回环、收敛、和多轮迭代。

### E1 正向推进链路

| 验收项 | 触发条件 | 通过标准 |
|--------|----------|----------|
| 需求→设计→验证→分解 一次通过 | 输入一个简单明确的需求 | 状态依次迁移，不回环，每次迁移的输入恰好是上一步的输出 |
| 分解→调度→执行→收集 无故障 | 所有工作包正常完成 | 调度尊重依赖顺序，结果被正确归一化，无工作包遗漏 |
| 缝合→产品验收→通过→终止 | 产出物满足所有验收标准 | 引擎输出"达到生产级别"判定并终止，不多跑一轮 |

### E2 回环正确性

| 验收项 | 触发条件 | 通过标准 |
|--------|----------|----------|
| 设计验证打回→产品设计 | 设计中发现域边界不清 | 回到产品设计，而非回到需求理解；修改后重新进入设计验证 |
| 产品验收打回→差距归因→精确回环 | 验收发现支付流程体验差 | 差距归因到设计层面→回迁到产品设计；归因到实现层面→回迁到任务分解重新出包 |
| 集成缝合失败→合约修复→重新实现 | 两个域的接口不兼容 | 接缝标记为 broken→合约修复并重新 freeze→仅重新调度受影响的工作包 |
| 失败恢复升级链 | 同一包连续失败 | 重试→切换执行器→重新分解→重新设计，升级链有序不跳级 |

### E3 多轮收敛

| 验收项 | 触发条件 | 通过标准 |
|--------|----------|----------|
| 差距递减 | 连续 3 轮产品验收 | 每轮发现的差距数量递减，而非震荡或增长 |
| 不做无效重复 | 同一差距第二次出现 | 引擎切换策略（换执行器、调粒度、改设计），而非用同样的方式重试 |
| 终止收敛 | 运行 N 轮后 | 在有限轮次内达到"生产级别"或触发人类升级，不无限循环 |
| 收敛失败兜底 | 连续 5 轮差距未收敛 | 引擎判定自主收敛失败，升级给人类，附上完整的迭代历史和每轮差距变化 |

### E4 并发与交错

| 验收项 | 触发条件 | 通过标准 |
|--------|----------|----------|
| 多执行器并行后汇合 | 3 个 claude_code 并行执行 3 个域 | 3 个结果全部收集后才进入缝合验证，不提前进入 |
| 并行中一个失败 | 3 个并行执行中 1 个失败 | 失败的进入重试流程，其余 2 个结果被保留，不因 1 个失败全部作废 |
| 需求变更打断执行中工作 | 3 个包在执行中，需求变更到达 | 评估影响：受影响的包被取消并重新规划，无关的包继续执行 |
| 跨项目交错调度 | 2 个子项目各有就绪的工作包 | 调度器按优先级交错调度，而非串行完成一个项目再做另一个 |

### E5 信息流完整性

| 验收项 | 触发条件 | 通过标准 |
|--------|----------|----------|
| 推送→拉取协作 | 执行器收到推送摘要后需要详情 | 推送中包含的引用 ID 能通过拉取接口查到完整内容，无断链 |
| 回环时知识更新 | 产品设计被修改后重新进入任务分解 | 新一轮工作包的上下文推送反映修改后的设计，而非缓存了旧版本 |
| 失败恢复时历史传递 | 工作包从 codex 切换到 claude_code | claude_code 能拉取到 codex 的失败记录和部分产出，不从零开始 |
| 多轮产出物累积 | 第 3 轮迭代的执行器 | 能拉取到前 2 轮所有已验证的产出物，而非只看到最近一轮 |

### E6 状态一致性

| 验收项 | 触发条件 | 通过标准 |
|--------|----------|----------|
| 回环后状态干净 | 从产品验收回迁到产品设计 | 受影响的工作包状态被正确回退（verified→deprecated），不存在"已验证但设计已变"的脏状态 |
| 崩溃恢复后状态一致 | 并发执行中引擎崩溃 | 恢复后 running 的包被标记为需重新调度，不出现"永久 running"的僵尸包 |
| 需求变更后状态一致 | 变更导致部分包 deprecated | deprecated 包不再出现在就绪队列中，依赖它的包被正确阻塞或重新评估 |

## 三、场景层验收

### 场景 S1：电商平台

**初始输入：**

> "做一个面向年轻人的二手交易平台，支持发布商品、搜索、下单、支付、评价，要有社区感。"

**验收检查点：**

| 轮次 | 预期状态迁移 | 检查内容 |
|------|-------------|----------|
| 第 1 轮 | 需求理解 | 引擎提炼出：目标用户（年轻人）、核心流程（发布→搜索→下单→支付→评价）、隐含需求（社区感＝用户互动、信任体系）、缺失信息（支付方式？配送还是自提？纠纷机制？）并向人类提问 |
| 第 2 轮 | 产品设计 | 输出域划分（用户、商品、交易、支付、评价、社区）、核心用户流程图、域间数据依赖、技术选型理由 |
| 第 3 轮 | 设计验证 | 验证发现：社区域和评价域职责重叠→打回设计→合并为"互动域"→重新验证通过 |
| 第 4 轮 | 任务分解 + 粒度控制 | 拆出约 15 个工作包，架构级（数据模型、API 骨架）分配 claude_code，页面实现分配 codex，每个包有明确验收标准 |
| 第 5 轮 | 调度 + 执行（批次 1） | 先调度无依赖的基础包（数据模型、认证）给 claude_code，3 个并行 |
| 第 6 轮 | 结果收集 + 缝合 | 3 个基础包产出收集，验证数据模型跨域一致 |
| 第 7 轮 | 调度 + 执行（批次 2） | 调度依赖基础包的业务包（商品发布、搜索、下单）给 codex，并行 |
| 第 8 轮 | 结果收集 + 缝合 | 故意注入故障：codex 的"下单"包失败。引擎重试→仍失败→切换到 claude_code 并调整粒度→成功 |
| 第 9 轮 | 调度 + 执行（批次 3） | 支付、评价、社区互动功能 |
| 第 10 轮 | 集成缝合验证 | 验证所有域产出物拼接正确，合约未破坏 |
| 第 11 轮 | 产品级验收（第 1 次） | LLM 以用户视角走读："发布→搜索→下单→支付→评价"。发现：搜索结果没有按距离/信任度排序，偏离"社区感"目标 |
| 第 12 轮 | 差距分析 | 归因：设计层面遗漏（搜索域未纳入社区信号）→回迁产品设计 |
| 第 13 轮 | 产品设计（局部修补） | 搜索域增加社区信号加权，只影响搜索相关工作包，其余不动 |
| 第 14 轮 | 重新分解 + 执行 | 新增 2 个工作包（搜索排序优化、社区信号接入），调度执行 |
| 第 15 轮 | 产品级验收（第 2 次） | 核心流程走通，社区感体现，达到生产级别→终止 |

**场景 S1 整体验收标准：**

- 引擎从模糊输入到终止，全程无人类干预（除第 1 轮的缺失信息确认）
- 经历至少 1 次设计回环、1 次执行器失败恢复、1 次产品验收打回
- 最终产出物被 LLM 判定为生产级别
- 已完成的无关工作包未被无谓重做

### 场景 S2：多人在线游戏

**初始输入：**

> "做一个 roguelike 地牢探索游戏，有随机地图、怪物战斗、装备掉落、角色成长，支持 2-4 人在线合作。"

**验收检查点：**

| 轮次 | 预期状态迁移 | 检查内容 |
|------|-------------|----------|
| 第 1 轮 | 需求理解 | 提炼核心循环（探索→战斗→掉落→成长→更深层）、多人需求（实时同步？回合制？）、缺失信息向人类提问 |
| 第 2 轮 | 产品设计 | 域划分（地图生成、战斗系统、物品经济、角色成长、多人同步、网络层），核心游戏循环设计 |
| 第 3 轮 | 设计验证 | 验证发现：多人同步和战斗系统耦合度过高→建议项目拆分 |
| 第 4 轮 | 多项目拆分 | 拆为 2 个子项目：单机核心玩法（地图+战斗+物品+成长）、多人联机层（同步+网络）。自动生成接缝合约（游戏状态同步接口、事件广播协议） |
| 第 5 轮 | 合约冻结 | 两个子项目的接缝合约进入 review→freeze，包含 schema、sample payload、集成测试用例 |
| 第 6 轮 | 并行执行子项目 1 批次 1 | 单机核心：地图生成 + 战斗系统基础（claude_code），并行 |
| 第 7 轮 | 并行执行子项目 2 批次 1 | 多人联机：网络传输层 + 状态同步框架（claude_code） |
| 第 8 轮 | 缝合验证（中期） | 验证两个子项目的产出物符合 frozen 合约。故意注入：子项目 1 的战斗事件格式偏离合约→seam 标记 broken |
| 第 9 轮 | 接缝修复 | 引擎定位偏离点，生成修复工作包，仅修复战斗事件格式，不重做整个战斗系统 |
| 第 10 轮 | 继续执行（批次 2） | 物品经济、角色成长、多人大厅 |
| 第 11 轮 | 需求变更注入 | 人类要求："加入 PvP 竞技场模式" |
| 第 12 轮 | 需求变更响应 | 影响评估：新增 PvP 域，影响战斗系统（需支持 PvP 规则）、多人同步（需匹配机制）、接缝合约需更新。已完成的地图生成、物品经济不受影响 |
| 第 13 轮 | 增量补丁 | 新增 PvP 相关工作包，更新受影响的合约，deprecated 的包仅限于与 PvP 冲突的部分 |
| 第 14 轮 | 继续执行 + 收集 | PvP 及修改后的工作包执行 |
| 第 15 轮 | 产品级验收（第 1 次） | LLM 以玩家视角评估：单人地牢探索循环是否有趣？多人合作是否流畅？PvP 是否平衡？发现：装备掉落概率影响 PvP 平衡性 |
| 第 16 轮 | 差距分析 + 回环 | 归因到产品设计层面（经济系统未考虑 PvP 平衡）→回迁产品设计局部修补→重新出包→执行 |
| 第 17 轮 | 产品级验收（第 2 次） | 核心循环完整、多人可用、PvP 可玩→生产级别→终止 |

**场景 S2 整体验收标准：**

- 成功完成项目拆分、合约冻结、并行执行、缝合验证全流程
- 经历至少 1 次合约破坏→修复、1 次中途需求变更、1 次产品验收回环
